# TOPIC 9: RequestDispatcher — Forward vs Include (vs Redirect)

## CONCEPT

### Why this concept exists

We've already used `RequestDispatcher.forward()` in Topic 1's MVC example. The Servlet API actually provides **two** distinct dispatching operations — `forward()` and `include()` — each with fundamentally different semantics and use cases. Additionally, `sendRedirect()` (Topic 8) is often confused with `forward()` since both "send the user somewhere else," despite being mechanically completely different.

### Real-world analogy

- **`forward()`** = You call a company's customer service line, get transferred internally to a specialist department — **you** (the caller) never know you were transferred; you just get your final answer from whoever handles it. One phone call, start to finish.
- **`include()`** = You call customer service, and they briefly patch in a translator to relay one specific part of the conversation, then **hand control back** to the original representative to continue and wrap up the call. Multiple contributors, one continuous call.
- **`sendRedirect()`** = The customer service representative says "I can't help with this, please hang up and call this other number instead" — **you** must dial again, a completely new phone call.

---

## `forward()` — Deep Formalization

### Mechanics (already traced in detail in Topic 1, Step 8)

```java
RequestDispatcher dispatcher = request.getRequestDispatcher("/WEB-INF/views/result.jsp");
dispatcher.forward(request, response);
```

**Precise rules:**
1. Control is **permanently transferred** to the target resource — any code **after** `dispatcher.forward(...)` in your original method still technically executes, but you should **never write response output after calling forward()** (see Common Errors below).
2. The **same** `request` and `response` objects are reused — all `request.setAttribute()` data remains accessible.
3. Happens entirely **server-side** — the browser is unaware, URL bar unchanged.
4. Can only forward to resources **within the same web application** (you cannot forward to an external website).
5. Response buffer: if the calling Servlet had already written **and flushed** content to the response before calling `forward()`, calling `forward()` afterward throws `IllegalStateException` — forwarding requires the response to be **uncommitted** (nothing flushed yet). This is why, in Topic 1's MVC example, the Servlet does validation/logic and calls `forward()` **without ever calling `getWriter()`** itself — all output generation is deliberately left entirely to the JSP.

## `include()` — New Formalization

### Mechanics

```java
RequestDispatcher dispatcher = request.getRequestDispatcher("/WEB-INF/views/header.jsp");
dispatcher.include(request, response);
// Execution CONTINUES here, back in the original Servlet, after the include finishes
out.println("Main content generated by the Servlet itself...");
```

**Precise rules:**
1. Control **temporarily** transfers to the target resource, which contributes **part** of the response, then control **returns** to the original caller, which can continue writing more content.
2. Multiple `include()` calls can be chained — e.g., include a header JSP, write your own main content, then include a footer JSP — assembling one response from multiple contributing pieces.
3. Same request/response objects shared, same request-scope attribute visibility.
4. The included resource **cannot** change the response's status code or commit headers in a way that affects the overall response — it's meant to contribute **body content only**, not control the overall response's metadata.

### Code Example — `include()` for a Reusable Header/Footer Pattern

**`WEB-INF/views/header.jsp`:**
```jsp
<html><body>
<div style="background:navy;color:white;padding:10px;">
    My Application — Header
</div>
```

**`WEB-INF/views/footer.jsp`:**
```jsp
<div style="background:gray;color:white;padding:10px;">
    &copy; 2026 My Company
</div>
</body></html>
```

**Servlet using `include()` to assemble a full page from three pieces:**

```java
@WebServlet("/dashboard")
public class DashboardServlet extends HttpServlet {

    private static final long serialVersionUID = 1L;

    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        response.setContentType("text/html");

        // Piece 1: include header
        request.getRequestDispatcher("/WEB-INF/views/header.jsp").include(request, response);

        // Piece 2: main content written directly by this Servlet
        PrintWriter out = response.getWriter();
        out.println("<h2>Welcome to your Dashboard</h2>");
        out.println("<p>Your data goes here.</p>");

        // Piece 3: include footer
        request.getRequestDispatcher("/WEB-INF/views/footer.jsp").include(request, response);
    }
}
```

**Notice the key structural difference from `forward()`:** the Servlet **itself** calls `getWriter()` and writes content **between** the two `include()` calls — something you'd **never** do with `forward()` (which expects to fully take over response generation). This is the clearest practical signal of the difference: **`forward()` = "someone else finishes this response." `include()` = "someone else contributes a piece, then I keep going."**

## `forward()` vs `include()` vs `sendRedirect()` — Definitive Comparison

| Aspect | `forward()` | `include()` | `sendRedirect()` |
|---|---|---|---|
| Where defined | `RequestDispatcher` | `RequestDispatcher` | `HttpServletResponse` |
| Number of HTTP requests | 1 | 1 | 2 (original + new one from browser) |
| Browser URL changes? | No | No | **Yes** |
| Request/response objects | Same, shared | Same, shared | New request entirely — old one discarded |
| Request-scoped attributes survive? | Yes | Yes | **No** (new request) |
| Control returns to caller? | No (target takes over fully) | **Yes** (caller resumes after) | N/A (browser makes new request) |
| Can target external sites? | No (same app only) | No (same app only) | **Yes** (any URL, internal or external) |
| Performance | Fast (single server-side hop) | Fast (single server-side hop) | Slower (extra network round-trip) |
| Typical use case | MVC: Controller → View | Reusable headers/footers, page composition | Post-login redirect, POST-then-redirect pattern (avoiding resubmission) |

## The "Post-Redirect-Get" Pattern (Practical Best Practice, Ties Everything Together)

This is a well-known industry pattern worth explicitly naming, since it combines everything from Topics 7, 8, and this topic:

**Problem:** If a user submits a form (POST), and your Servlet responds by directly writing/forwarding to a "success" page, then hits **browser refresh**, the browser will **re-submit the exact same POST request** (triggering the browser's "Resubmit form?" warning) — potentially creating a duplicate action (e.g., double-charging a payment, double-creating a record).

**Solution — Post-Redirect-Get (PRG):**
```java
// After successfully processing a POST:
session.setAttribute("successMessage", "Login successful!"); // temporarily, session scope
response.sendRedirect("dashboard"); // browser makes a NEW GET request
```
Now, refreshing the dashboard page just **re-issues the harmless GET** request, not the original POST — no duplicate-submission risk. This is exactly why Topic 7's login example used `sendRedirect()` after successful login rather than `forward()` — you now understand precisely *why* that design choice was made, not just that it "works."

```
                              │
                              ▼
                    Response #2 sent to Browser
                    (Browser URL now shows /dashboard)
```

## COMMON ERRORS

**Error: `IllegalStateException: Cannot forward after response has been committed`**
```java
out.println("Some content...");
response.getWriter().flush(); // commits the response
dispatcher.forward(request, response); // FAILS — too late
```
- **Cause:** Attempting to `forward()` after the response buffer was already flushed/committed.
- **Fix:** Never write to the response in a Servlet that intends to `forward()` afterward — let the target resource handle all output generation (exactly the discipline shown in Topic 1's MVC example).

**Error: Blank or incomplete page when misusing `include()`**
- **Cause:** Forgetting that included resources shouldn't try to set the overall page's status/headers, or forgetting the surrounding HTML structure (e.g., `header.jsp` opens `<html><body>` but if you forget the matching close tags in `footer.jsp`, you get malformed HTML) — as shown in the example above, this "split HTML across files" pattern requires careful discipline about where tags open/close.
- **Fix:** Plan the HTML structure across all included fragments together, ensuring tags are properly balanced across the full assembled response.

**Error: Expecting request-scoped data to survive a redirect (repeated from Topic 7, worth reinforcing here)**
- **Fix:** Use session scope for data that must survive a redirect (the PRG pattern above), and remember to remove it from the session immediately after reading it on the next request, to avoid stale data lingering.
