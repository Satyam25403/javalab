# TOPIC 1 (continued): Project Structure for Servlet-Based Applications

## STEP 3 — PROJECT STRUCTURE

Before writing any Servlet code, you must understand **exactly** how a Java web application is organized on disk, because the Servlet Container (Tomcat) has **strict, non-negotiable expectations** about where files live. If you place a file in the wrong folder, your application simply won't work — no error message will always make this obvious to a beginner, so understanding this deeply now saves you hours of debugging later.

### Complete Folder Structure (Maven-based, Modern Standard)

This is the structure we will use throughout this course, since Maven is the industry standard build tool:

```
MyWebApp/                              (Project root / Maven project folder)
│
├── pom.xml                            (Maven Project Object Model — dependencies, build config)
│
├── src/
│   ├── main/
│   │   ├── java/                      (All your .java source files go here)
│   │   │   └── com/
│   │   │       └── company/
│   │   │           └── myapp/
│   │   │               ├── servlet/
│   │   │               │   └── HelloServlet.java
│   │   │               ├── dao/
│   │   │               │   └── UserDAO.java
│   │   │               ├── model/
│   │   │               │   └── User.java
│   │   │               ├── service/
│   │   │               │   └── UserService.java
│   │   │               ├── filter/
│   │   │               │   └── AuthFilter.java
│   │   │               ├── listener/
│   │   │               │   └── AppContextListener.java
│   │   │               └── util/
│   │   │                   └── DBUtil.java
│   │   │
│   │   ├── resources/                 (Non-Java files needed at runtime — config files, etc.)
│   │   │   └── db.properties
│   │   │
│   │   └── webapp/                    (Root of the deployable web application — THE MOST IMPORTANT FOLDER)
│   │       ├── WEB-INF/
│   │       │   ├── web.xml            (Deployment Descriptor — servlet mappings, config)
│   │       │   ├── classes/           (Compiled .class files land here after build — auto-generated)
│   │       │   └── lib/                (JAR dependencies — auto-populated by Maven build)
│   │       │
│   │       ├── index.jsp              (Welcome/landing page)
│   │       ├── css/
│   │       │   └── style.css
│   │       ├── js/
│   │       │   └── script.js
│   │       ├── images/
│   │       │   └── logo.png
│   │       └── views/                 (JSP pages — kept OUTSIDE direct browser access sometimes)
│   │           └── userList.jsp
│   │
│   └── test/
│       └── java/                      (Unit test classes — JUnit tests, covered later)
│
└── target/                            (Build output folder — auto-generated by Maven, contains the final .war file)
```

### Why Each Folder Exists — Deep Explanation

**1. `pom.xml` (Project root)**
This is the **heart of a Maven project**. It declares:
- Project metadata (group ID, artifact ID, version)
- Dependencies (Servlet API, JSP API, JSTL, MySQL Connector, etc.)
- Build plugins (how to compile, package into a `.war` file)
- Packaging type (`<packaging>war</packaging>` — critical for web apps, tells Maven to produce a WAR, not a plain JAR)

We cover this exhaustively in Step 4 (Dependency Setup).

**2. `src/main/java/`**
This is where **all your Java source code** lives — Servlets, DAOs, Models, Services, Filters, Listeners, Utility classes. Maven's standard directory layout convention **mandates** this exact path; deviating from it requires custom configuration in `pom.xml`, which is not recommended for beginners.

*Why organize into sub-packages (`servlet`, `dao`, `model`, `service`)?* This reflects **layered architecture** / **separation of concerns** — a core software engineering principle you'll rely on heavily once we reach MVC and later Spring:
- `servlet/` → Controller layer (handles HTTP requests, delegates to service layer)
- `service/` → Business logic layer (validation, business rules)
- `dao/` → Data Access layer (talks to the database via JDBC)
- `model/` → Plain Java Objects (POJOs) representing your data (e.g., `User`, `Employee`)
- `filter/` → Cross-cutting concerns (authentication, logging) that intercept requests before/after Servlets
- `listener/` → Objects that react to application-level or session-level lifecycle events
- `util/` → Reusable helper classes (e.g., database connection utilities)

**3. `src/main/resources/`**
Any **non-Java files** that your Java code needs to **read at runtime** (like `.properties` files for database configuration, log4j configs, etc.) go here. Maven automatically copies everything in this folder into `WEB-INF/classes/` during the build — this matters because Java's classpath-based resource loading (`getResourceAsStream()`) looks inside `WEB-INF/classes/` at runtime.

**4. `src/main/webapp/` — THE MOST CRITICAL FOLDER**
This entire folder becomes the **root of your deployed web application**. Everything inside it — except `WEB-INF` — is **directly and publicly accessible via URL** from the browser.

For example, if your app is deployed as context path `/myapp`, then:
- `webapp/index.jsp` → accessible at `http://localhost:8080/myapp/index.jsp`
- `webapp/css/style.css` → accessible at `http://localhost:8080/myapp/css/style.css`

**5. `WEB-INF/` — The Protected Folder**
This is a **special, container-recognized folder name** (mandated by the Servlet specification itself — you cannot rename it). Anything inside `WEB-INF/` is **NOT directly accessible via a browser URL**, even though it physically sits inside `webapp/`. If a user types `http://localhost:8080/myapp/WEB-INF/web.xml` directly into their browser, Tomcat will return a **404 Not Found**, by design — this is a **security mechanism**.

*Why does this matter?* You don't want users directly accessing your compiled `.class` files, your `web.xml` configuration (which may reveal internal servlet mappings), or — critically — your JSP files if you're following a proper MVC pattern where JSPs should only be reachable **through** a Servlet (via `RequestDispatcher.forward()`), never directly by URL. This is precisely why many real projects place JSPs inside `WEB-INF/views/` instead of directly under `webapp/` — forcing all requests to go through a Controller Servlet first, enforcing proper MVC flow. (We'll implement this exact pattern in your MVC project later in Module 3.)

**6. `WEB-INF/web.xml`**
The **Deployment Descriptor** — an XML file where you can declare Servlets, their URL mappings, initialization parameters, session timeout, error pages, filters, listeners, and more — as an **alternative to annotations**. Fully explained tag-by-tag in Step 5.

**7. `WEB-INF/classes/`**
This is where your **compiled `.class` files** must reside at runtime, mirroring your Java package structure exactly (e.g., `com/company/myapp/servlet/HelloServlet.class`). When using Maven, you never manually place files here — the build process does it automatically by compiling everything in `src/main/java/` and copying the output here during packaging.

**8. `WEB-INF/lib/`**
This is where all **JAR dependency files** must be placed at runtime (e.g., MySQL Connector JAR, JSTL JAR) so that Tomcat's WebappClassLoader can find and load those classes. Again, Maven automates this — it copies all your declared `<dependency>` JARs here automatically during the build, based on your `pom.xml`.

**9. `target/`**
Auto-generated by Maven when you run a build command (`mvn package`). Contains the compiled classes, and ultimately, the packaged **`.war` (Web Application Archive)** file — this is the single file you actually deploy to Tomcat (by placing it in Tomcat's `webapps/` folder, covered in Step 4).

### Naming Conventions

| Item | Convention | Example |
|---|---|---|
| Package names | all lowercase, reverse domain style | `com.company.myapp.servlet` |
| Class names | PascalCase | `HelloServlet`, `UserDAO` |
| Servlet class names | End with "Servlet" (convention, not mandatory) | `LoginServlet`, `RegisterServlet` |
| JSP file names | lowercase or camelCase, `.jsp` extension | `userList.jsp`, `login.jsp` |
| URL patterns | lowercase, often REST-like even in old-style apps | `/login`, `/user-list` |
| Properties files | lowercase, `.properties` extension | `db.properties` |

### Where Every File Type Should Be Placed — Quick Reference Table

| File Type | Correct Location |
|---|---|
| `.java` source files | `src/main/java/<package>/` |
| Compiled `.class` files | `WEB-INF/classes/` (auto-generated, never manual) |
| `.jsp` files (MVC pattern) | `WEB-INF/views/` (not directly browser-accessible) |
| `.jsp` files (simple apps) | Directly under `webapp/` |
| `web.xml` | `WEB-INF/web.xml` (exact name, exact location — mandatory) |
| Third-party `.jar` files | `WEB-INF/lib/` (auto-managed by Maven) |
| Static assets (CSS, JS, images) | `webapp/css/`, `webapp/js/`, `webapp/images/` |
| `.properties` config files | `src/main/resources/` |
